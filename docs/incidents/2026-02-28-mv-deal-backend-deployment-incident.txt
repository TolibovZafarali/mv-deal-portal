Incident Note: mv-deal-backend Cloud Build/Cloud Run Deployment Failures
Date: February 28, 2026 (UTC)
Project: megna-real-estate-backend
Service: mv-deal-backend
Region: us-central1
Primary build failure step: deploy-cloud-run (Cloud Build step 3)

1) Executive Summary
Deployments repeatedly failed at Cloud Run revision startup, while image build and push steps succeeded.
This incident had multiple sequential root causes:
1. Java runtime dependency conflict (protobuf / Cloud SQL connector) causing java.lang.VerifyError.
2. Flyway migration validation failure (V9 marked failed in flyway_schema_history).
3. Schema drift (property_photos.photo_asset_id missing), causing Hibernate schema validation failure at startup.

Service was stabilized temporarily by disabling Flyway and JPA schema checks, then permanently fixed by repairing schema/history and restoring strict validation settings.

2) Service Impact
- Existing stable revision continued serving traffic for part of the incident while new revisions failed.
- New backend releases could not be deployed during the incident window.
- Schema drift risk existed until DB and Flyway history were corrected.

4) Root Causes
A) Runtime dependency mismatch:
- Cloud SQL socket factory path loaded incompatible protobuf runtime classes, causing VerifyError during DB pool initialization.

B) Failed Flyway migration state:
- flyway_schema_history had V9__add_photo_assets_and_migrate_property_photos.sql with success=0.

C) Database schema drift:
- property_photos lacked expected columns used by the application model (photo_asset_id and thumbnail_url).

5) Fixes Applied
- Adjusted backend dependency handling in pom.xml to address protobuf/runtime conflict in the MySQL/Cloud SQL path.

- Temporary Cloud Run stabilization env vars:
  - SPRING_FLYWAY_ENABLED=false
  - SPRING_FLYWAY_VALIDATE_ON_MIGRATE=false
  - SPRING_JPA_HIBERNATE_DDL_AUTO=none

- Manual DB repair (database: megnarealestate):
  - Added column thumbnail_url to property_photos.
  - Added column photo_asset_id to property_photos.
  - Backfilled where applicable.
  - Enforced photo_asset_id as NOT NULL.
  - Verified zero rows with missing photo_asset_id.

- Flyway history correction:
  - Updated flyway_schema_history version 9 row from success=0 to success=1.

- Restored strict runtime settings:
  - SPRING_JPA_HIBERNATE_DDL_AUTO=validate
  - SPRING_FLYWAY_ENABLED=true
  - SPRING_FLYWAY_VALIDATE_ON_MIGRATE=true

6) Final Verification
- gcloud run services describe confirmed latest ready revision equals latest created revision:

- Health check:

  - Response: HTTP/2 200, status UP

7) Why Troubleshooting Took Time
- Cloud Build surfaced only generic startup probe failures.
- Root causes were in Cloud Run stdout/stderr logs.
- gcloud intermittently crashed with TypeError (string indices must be integers), requiring Logging API fallback.
- Multiple independent failures appeared sequentially, so each exposed the next issue only after the previous one was resolved.

8) Prevention Actions
1. Add CI dependency checks for protobuf/Cloud SQL connector compatibility.
2. Add pre-deploy migration validation against a prod-like MySQL environment.
3. Avoid manual edits to flyway_schema_history except incident-approved break-glass steps with documentation.
4. Add no-traffic revision health gate before routing 100% traffic.
5. Create and retain Cloud SQL backup snapshots around each production schema-change release.
